Advanced Python Interview Guide

A deep-dive reference for senior Python developer interviews at latency-sensitive firms. Concepts, internals, and interview-ready Q&A.

1. Orientation — How to Use This Guide

This guide is tuned for senior Python developer interviews at latency-sensitive engineering firms: a technically demanding mix of Python internals, systems thinking, and low-latency engineering sensibility. It assumes you can already write Python fluently and focuses on the kind of why questions that come up once the interviewer probes past syntax.

Each section follows the same shape:

  • Concept — a tight explanation of the underlying mechanism in CPython, with code that you could write on a whiteboard.
  • Talking points — one-liners you can drop into an answer to signal depth without monologuing.
  • Interview Q&A — representative questions with model answers. A final "Mixed" section cross-cuts topics the way a real interviewer will.
How to study this
Read each section once top-to-bottom. Then do a second pass where you cover the answers in the Q&A boxes and talk through each one out loud — you will catch gaps the eye skipped over. For low-latency topics, always have a concrete number in your head (cache line size, GIL switch interval, dict load factor, etc.). Interviewers reward specificity.

What the interviewer actually cares about

At a latency-sensitive firm, Python is typically the glue around a C++ trading or infrastructure core — not a data-science role. Expect questions that probe:

  • Where Python is slow and how to get around it — the GIL, interpretation overhead, allocation costs, and the escape hatches (NumPy, Cython, pybind11, numba, shared memory).
  • Memory model awareness — reference counting, cyclic GC, __slots__, object layout, why sys.getsizeof lies, zero-copy with memoryview and the buffer protocol.
  • Concurrency literacy — when to reach for threads, asyncio, multiprocessing, and when none of the above are the right answer.
  • Production discipline — typing, packaging, profiling, testing, observability. They want someone who can ship Python that doesn't become the bottleneck in a trading day.

2. The Python Execution Model

Before memory, concurrency, or performance make sense, you need a clean mental model of what actually happens when the interpreter runs your code. Python is neither "purely interpreted" nor "purely compiled" — it is compiled to bytecode and then that bytecode is executed by an interpreter loop.

Source → AST → Bytecode → Frame → ceval

Given a .py file, CPython walks through roughly the following stages the first time the module is imported:

  1. Tokenize / parse the source into a concrete syntax tree, then into an AST (the ast module exposes this).
  2. Compile the AST into a code object: a block of immutable bytecode plus associated constants, names, free variables, line-number table, and exception-handler table.
  3. Cache the code object as a .pyc file under __pycache__/, keyed by interpreter version and source hash/mtime, so subsequent imports skip parsing.
  4. At call time, the interpreter builds a frame object (one per function invocation): it holds the local variable array, value stack, code object reference, and caller pointer. Frames live on the heap, linked through f_back — this is Python's call stack.
  5. The core evaluation loop (ceval.c's giant switch / computed-goto dispatch) iterates through the bytecode of the current frame, pushing and popping values on the frame's value stack.

You can inspect every piece of this from pure Python:

import dis, ast

src = """
def add(a, b):
    return a + b
"""

tree = ast.parse(src)                 # AST
code = compile(tree, "<demo>", "exec") # code object
dis.dis(code)                         # disassemble bytecode

The output for add on CPython 3.12+ looks roughly like: 

  2           0 RESUME                   0

  3           2 LOAD_FAST                0 (a)
              4 LOAD_FAST                1 (b)
              6 BINARY_OP                0 (+)
             10 RETURN_VALUE

Every bytecode instruction is a single byte of opcode plus an argument byte (historically; modern CPython uses wordcode — 2-byte units with EXTENDED_ARG for wider operands). ceval reads each instruction and performs the corresponding action on the value stack. That is all "the Python interpreter" is at the lowest level.

Code objects are immutable and sharable

A code object is essentially compiled IR. It does not capture any runtime state — that lives in the frame. This is why you can pickle bytecode, why the same function body can be reused across instances with different closures, and why decorators can introspect __code__.co_varnames, co_consts, co_names, etc. 

def make_counter(start):
    def inc():
        nonlocal start
        start += 1
        return start
    return inc

inc = make_counter(10)
inc.__code__.co_freevars   # ('start',) — captured by closure
inc.__closure__            # tuple of cell objects holding the actual values

The specializing adaptive interpreter (PEP 659, 3.11+)

Since 3.11, CPython watches hot bytecode and specializes instructions based on observed types. A generic BINARY_OP might get rewritten in-place to BINARY_OP_ADD_INT after seeing a few int-int additions, skipping the dispatch through __add__. If a later call sees a float, it deoptimizes back to the generic form. This is why 3.11+ is noticeably faster than 3.10 on real workloads without any JIT involvement.

Talking point
"CPython is not a JIT, but since 3.11 it's a specializing interpreter. Hot instructions mutate into type-specialized variants in-place — the compiler is still ahead-of-time but the dispatch loop adapts. 3.13 is layering an experimental copy-and-patch JIT on top of this."

Startup and import: what .pyc actually saves

On import, Python computes the expected .pyc path and checks the magic number and source metadata. On a hit it loads the marshaled code object directly — no parser, no compiler. On a miss it goes through the full pipeline and writes the new .pyc. .pyc files do not contain machine code — they contain pickled bytecode. Startup cost in Python is overwhelmingly dominated by the number of modules imported, not by parsing.

Exception handling: zero-cost try, expensive except

Since 3.11, try blocks have effectively zero runtime cost — there is no SETUP_EXCEPT bytecode any more. Instead, each code object carries a compact exception table that maps bytecode offsets to handler offsets. Entering the try does nothing; on an exception the interpreter does a table lookup to find where to jump. That makes try/except cheap for the success path but exceptions themselves (with traceback construction) are still expensive. Never use exceptions as control flow in a hot loop.

Interactive probing — the tools you want muscle memory for

import dis, sys, inspect

dis.dis(func)                 # disassemble bytecode
dis.show_code(func)           # code object metadata
sys.getsizeof(x)              # shallow size of an object
sys.getrefcount(x)            # reference count (+1 for the arg itself)
inspect.getsource(func)       # source, if available
inspect.signature(func)       # parameter inspection
func.__code__.co_varnames     # local variable names
func.__code__.co_consts       # constants captured at compile time

Walk me through what happens from the moment I run python script.py until my first line executes.

The interpreter binary initializes the runtime, sets up the GIL and the main thread state, imports a small set of built-in modules (sys, builtins, etc.), and then invokes the importer on the target file. For that file it tokenizes, parses to an AST, compiles to a code object, and caches a .pyc in __pycache__. It then builds a module object, assigns it to sys.modules['__main__'], creates a frame backed by that code object and the module's globals, and hands that frame to ceval, which starts dispatching bytecode. Every top-level statement runs as bytecode in that module frame; def and class are themselves just bytecode that builds function and class objects.

Is Python interpreted or compiled?

Both, in different senses. The source is compiled ahead of time to bytecode — the code object is a real compilation artifact, cached on disk as .pyc. But the bytecode is executed by a software interpreter (ceval) rather than by the CPU directly. Since 3.11, that interpreter specializes hot instructions based on observed types, and 3.13 ships an experimental copy-and-patch JIT behind a build flag. PyPy takes this all the way and traces and JITs Python directly to machine code.

Why is try/except basically free but raising an exception still costly?

Since 3.11 there is no bytecode to set up a try block — each code object carries a compact exception table mapping bytecode ranges to handler offsets. Entering the try emits no instruction at all. But when an exception is raised, CPython still constructs the exception object, walks the frames to build a traceback (allocating frame snapshots), and does the table lookup. Profiling shows raising a simple exception is hundreds of nanoseconds to a couple of microseconds. It's fine for truly exceptional paths; it's terrible as control flow in a hot loop. Use sentinels or explicit checks there.

3. The Object & Data Model

Python's object model is the piece that unlocks most of the "how does this actually work" questions. Once you internalize that everything is a PyObject* and every operation on it goes through a type's slot table, a lot of behavior that looks magical becomes mechanical.

Every value is a heap-allocated PyObject

A CPython object, at the C level, has a header with (at minimum) a reference count and a type pointer: 

typedef struct _object {
    Py_ssize_t ob_refcnt;     // reference count
    PyTypeObject *ob_type;    // type, aka the "class"
} PyObject;

Everything else — attribute dictionaries, values, lengths, iterator state — is tacked on after that header depending on the type. There is no stack-allocated primitive. Even x = 3 allocates (or retrieves a cached) PyLongObject on the heap and binds the name x to a pointer to it. This is the single biggest reason Python is slow in tight numeric loops: every operation chases pointers through headers, and every intermediate result is a fresh heap allocation.

Types are themselves objects

type is both the built-in that creates classes and the metaclass of all built-in classes. type(obj) returns the type, which is itself an instance of type: 

type(3)           # <class 'int'>
type(int)         # <class 'type'>
type(type)        # <class 'type'>   (it is its own metaclass)
isinstance(int, type)  # True

The dunder protocol is the slot table

When you write a + b, CPython doesn't call a Python method — it looks at type(a)->tp_as_number->nb_add, a C slot in the type struct. For pure-Python classes, that slot points to a trampoline that dispatches to __add__. This is why dunder methods are looked up on the type, not on the instance: 

class C:
    def __len__(self): return 5

c = C()
c.__len__ = lambda: 99    # instance attribute, ignored by len()
len(c)                    # 5 — len() hits type(c).__len__

The dunders you need at your fingertips:

Attribute lookup — the full algorithm

When you write obj.x, CPython runs an algorithm that trips up a lot of candidates. Memorize it:

  1. Look up x on type(obj) walking the MRO. If found and it's a data descriptor (defines __set__ or __delete__), call its __get__ and return that.
  2. Otherwise look up x in obj.__dict__ (or __slots__ storage). If found, return it.
  3. Otherwise, if the earlier class-level lookup found a non-data descriptor (only __get__, e.g., a function / @staticmethod / @classmethod), call its __get__.
  4. If still nothing, call type(obj).__getattr__(obj, 'x') if defined.
  5. Raise AttributeError.

The critical asymmetry: data descriptors override instance dicts, non-data descriptors do not. That is what makes @property immune to instance shadowing and what makes plain methods overridable by setting an attribute on the instance.

Descriptors — how @property, classmethod, staticmethod all work 

class LoggedAttr:
    def __set_name__(self, owner, name):
        self.name = name
    def __get__(self, obj, objtype=None):
        if obj is None: return self
        print(f"GET {self.name}")
        return obj.__dict__[self.name]
    def __set__(self, obj, value):
        print(f"SET {self.name} = {value}")
        obj.__dict__[self.name] = value

class Order:
    price = LoggedAttr()

o = Order()
o.price = 101.25   # SET price = 101.25
o.price            # GET price

@property, functools.cached_property, classmethod, staticmethod, and the machinery behind bound methods are all descriptors. cached_property is a non-data descriptor — that's how it stores the computed value in the instance dict and shadows itself on subsequent accesses.

MRO and C3 linearization

With multiple inheritance, the method resolution order is computed using the C3 algorithm. You can inspect it with Cls.__mro__ or Cls.mro(). The rules you need to state: C3 preserves the local precedence order of each parent list, preserves monotonicity (a subclass's MRO is a linear extension of each parent's MRO), and raises TypeError at class-creation time if no consistent order exists. 

class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass
D.__mro__   # (D, B, C, A, object)

super() walks the MRO from the current class forward — not "to the parent." That's why cooperative multiple inheritance works: every class in a diamond gets called exactly once as long as each calls super().method(...).

__slots__ — cheap and fast instances

By default, instance attributes live in a per-instance dict. With __slots__, CPython reserves a fixed array of descriptors on the class instead. This:

  • Saves memory (no per-instance dict — 100+ bytes per instance at minimum).
  • Speeds up attribute access (no hash, just an index).
  • Prevents accidental attribute creation (great for bug-prevention in data-model classes). 
class Tick:
    __slots__ = ("ts", "bid", "ask")
    def __init__(self, ts, bid, ask):
        self.ts, self.bid, self.ask = ts, bid, ask

import sys
sys.getsizeof(Tick(0,1,2))     # ~48 bytes vs ~152 for __dict__ equivalent

Gotchas: subclasses need their own __slots__ (or they grow a dict); __slots__ plus multiple inheritance with non-empty slot classes is restricted; you lose __weakref__ unless you include it. In tight market-data code paths, __slots__ is essentially free performance.

Quant relevance
A trading signal that processes 10M ticks per minute over Python objects is going to spend significant time in allocator and attribute-lookup code. Slotted classes and NumPy structured arrays / record arrays are the two canonical tools for collapsing that overhead.

What's the difference between __new__ and __init__?

__new__ is a static method on the type that allocates and returns the instance — it's where you'd intercept construction for singletons, interning, or subclassing immutable types like tuple or int. __init__ takes the already-constructed instance and initializes it in place; it must return None. You override __new__ when you need to actually change what object is returned; you override __init__ when you just want to set state.

I set obj.method = lambda self: ... and it didn't behave as a method. Why?

Functions become bound methods via the descriptor protocol — specifically, the non-data descriptor __get__ on a function object returns a MethodType that binds self. That only happens when the function is found on the type, not on the instance. A function assigned directly to obj.method is just an attribute; calling obj.method() calls it with zero arguments. Assign it to the class (or wrap it in types.MethodType(func, obj)) if you want binding.

When should I use __slots__?

When you have many instances of a small fixed-shape class and memory or attribute-access speed matters. Typical candidates: tick records, order-book levels, graph nodes, cache entries. Don't reach for it on every class — you lose __dict__, inheritance gets fiddly, and it complicates serialization. And always measure: sometimes a @dataclass(slots=True) or a NumPy structured array is the better answer.

4. Memory Management

CPython's memory story is deceptively simple: reference counting for prompt reclamation, plus a generational cycle collector for the edge cases refcounting can't handle, on top of a specialized small-object allocator (pymalloc). Understanding the interaction between these three pieces is what the advanced questions probe.

Reference counting is the primary mechanism

Every PyObject header starts with a Py_ssize_t ob_refcnt. Every assignment, argument pass, return, container insertion, and closure capture increments or decrements that count via Py_INCREF / Py_DECREF. When the count hits zero, the object is deallocated immediately — its type's tp_dealloc runs, it decrefs any children (recursively freeing them), and returns the memory to pymalloc. 

import sys

x = [1, 2, 3]
sys.getrefcount(x)   # 2: one for `x`, one for the argument binding
y = x
sys.getrefcount(x)   # 3
del y
sys.getrefcount(x)   # 2

Consequences you should be able to cite on the spot:

  • Deterministic destruction: objects are usually freed the instant the last reference goes out of scope. This is why with open(...) as f: works naturally even without the context manager — though you should always use it for clarity and safety under exceptions.
  • No stop-the-world for most work — unlike tracing GCs, refcounting spreads the cost across the program. But it makes every pointer assignment more expensive than in a refcount-free runtime.
  • Cycles leak without help. a.child = b; b.parent = a would never reach zero without the cycle collector.
  • The refcount field is the reason the GIL exists in the way it does — Py_INCREF/Py_DECREF are not atomic by default, and making them atomic has a measurable cost. PEP 703 (no-GIL / free threading) makes them atomic with biased reference counting to recover performance.

The generational cycle collector

The gc module layers a mark-and-sweep cycle collector on top of refcounting. Only container types (those whose type struct has tp_traverse defined) are tracked — integers, floats, strings, and simple tuples of only non-container objects aren't. That's both why GC is relatively cheap and why you can't just disable it for latency: the objects being tracked are exactly the ones most likely to form cycles.

Generations work like most tracing GCs: new objects go into generation 0, survive into 1, then 2. Thresholds on the number of allocations-minus-deallocations trigger collection: 

import gc
gc.get_threshold()   # (700, 10, 10) default: gen0 every ~700 allocs, gen1 every 10 gen0 sweeps, gen2 every 10 gen1 sweeps
gc.get_count()       # live counts per generation
gc.collect(generation=2)
gc.disable()         # caveat emptor — cycles will leak until re-enabled
Latency gotcha
A gen2 collection on a large long-lived heap can stall for milliseconds to tens of milliseconds. In latency-sensitive Python tools it is a common trick to gc.disable() during a hot path and re-enable it (or gc.collect() explicitly) at a known quiescent point. If you do this, you must be sure your code does not create long-lived cycles or you will OOM.

The private heap and pymalloc

CPython does not call malloc for every object. Instead, allocations smaller than 512 bytes go through pymalloc, a pool allocator:

  • Arenas are 256 KB slabs of memory obtained from the system allocator.
  • Each arena is carved into 4 KB pools, and each pool serves a single size class (multiples of 16 bytes).
  • Each pool is carved into fixed-size blocks which are the actual objects.

The upshot: allocating a small Python object is typically a push onto a per-size-class free list — very fast, and it keeps same-sized objects close together in memory (which matters a lot more than people think for cache behavior). Allocations larger than 512 bytes fall through to the system allocator.

You can turn on allocation debugging and inspect the internal arenas: 

import sys, tracemalloc

tracemalloc.start()
# ... run code ...
snap = tracemalloc.take_snapshot()
for stat in snap.statistics('lineno')[:10]:
    print(stat)

Caches, singletons, and interning

CPython caches a handful of immutable values to avoid allocating new objects for common cases. The ones interviewers love to ask about:

  • Small ints cache — integers in [-5, 256] are preallocated singletons. This is why 256 is 256 is True in every context but 257 is 257 may or may not be, depending on whether the literals were compiled into the same code object (where the compiler may intern them) vs constructed at runtime.
  • Interned strings — identifiers and short strings that look like identifiers are automatically interned. You can force interning with sys.intern(s), which is useful when you store lots of string keys and want fast == and hash.
  • Empty tuple, empty bytes, empty frozenset are singletons.
  • None, True, False are singletons — that's why x is None is both idiomatic and correct. 
a = 256; b = 256;  a is b    # True
a = 257; b = 257;  a is b    # False (usually)
import sys
sys.intern("long-identifier-like-string") is sys.intern("long-identifier-like-string")  # True

Weak references

Normal references keep objects alive. weakref lets you hold a pointer that does not bump the refcount. Perfect for caches, observer patterns, and parent-pointers that would otherwise form cycles. 

import weakref

class Book: pass
b = Book()
r = weakref.ref(b)
r() is b                 # True — dereference
del b
r()                      # None — object has been collected

Not every type supports weakrefs — int, str, tuple, and list don't (no __weakref__ slot). Classes with __slots__ need to explicitly include __weakref__ in the slot list to support them.

sys.getsizeof lies to you (kind of)

getsizeof returns the shallow size — it includes the object's header and its own directly allocated storage, but not the things it references. For a list of dicts you need to walk the structure yourself, or use pympler.asizeof. 

import sys
sys.getsizeof([])         # ~56 bytes — the empty list object itself
sys.getsizeof([1,2,3])    # ~88 bytes — header + three pointers, NOT including the ints
sys.getsizeof("")         # ~49
sys.getsizeof({})         # ~64 — empty dict has a special compact form

Buffer protocol, memoryview, and zero-copy

The buffer protocol is how types like bytes, bytearray, array.array, and NumPy arrays expose a pointer into raw memory. memoryview is the Python-level wrapper. It lets you slice, reshape, and hand buffers to C code without copying, which is critical for any code that touches market data frames or shared memory. 

buf = bytearray(1024)
mv = memoryview(buf)
header, body = mv[:8], mv[8:]   # zero-copy slicing
body.cast('d')[0] = 3.14        # interpret as doubles, in place

__del__ and finalizers — just don't

Override __del__ only with extreme caution. It runs at an unpredictable time, can resurrect its object, suppresses exceptions, and interacts badly with cycles. Use weakref.finalize or a context manager instead. If you're tempted to use __del__ to close a socket or file handle, write a context manager — your future self will thank you.

Interview one-liner
"CPython uses refcounting for prompt destruction and a generational cycle collector for the remainder. Refcounting is what makes RAII-style resource management work without GC pauses, and it's also what forced the GIL's specific design."

Why does Python need both reference counting and a garbage collector?

Refcounting handles 99% of cleanup promptly and cheaply — as soon as the last reference goes away, the object is freed. But refcounting can't handle reference cycles: two objects pointing at each other will never hit zero. The generational cycle collector exists solely to find and break those cycles. It only tracks container types that can participate in cycles, and it runs periodically based on allocation thresholds. You can disable it with gc.disable() in a hot path if you can guarantee no cycles.

What does __slots__ do to memory, and when does it hurt?

A normal instance stores attributes in a per-instance __dict__, which adds roughly 100+ bytes of overhead and does a hash lookup on every access. __slots__ replaces that with a fixed array of C-level descriptors on the class, so attribute access becomes an array index and the instance loses its dict entirely. Memory savings are substantial at scale. The costs: you can't add arbitrary attributes, subclassing has to be careful (subclasses need their own slots or they regrow a dict), multiple inheritance with non-empty slots is restricted, and you lose weakref support unless you opt in. For a market-data Tick class you instantiate millions of times, __slots__ is free money.

Explain the difference between shallow and deep copy, and when you need each.

copy.copy constructs a new container with the same element references — mutating a nested element affects both copies. copy.deepcopy recursively reconstructs the entire object graph and correctly handles cycles via a memo dict. Shallow copy is the right default when elements are immutable or when you want shared subobjects; deep copy is for fully independent snapshots. Deep copy is expensive: it pickles semantics and walks the whole graph, so in hot paths you usually want a type-specific clone (list(x), dict(x), np.copy(arr)) rather than the generic function.

I noticed my Python process's RSS grows after a big workload and doesn't shrink. Is that a leak?

Probably not in the CPython sense — it's usually allocator fragmentation. pymalloc can release an entire arena back to the OS only when every block inside it is free, and long-lived objects scattered across arenas prevent that. You can confirm with tracemalloc or gc.get_objects() that the Python-visible object count is stable. Real leaks usually show up as unbounded growth in a specific allocation site. Fixes: avoid long-lived small objects mixed with short-lived ones, use __slots__, or, for worker pools, recycle subprocesses after N tasks.

5. Concurrency, Parallelism, and the GIL

This is the single most asked-about topic in Python interviews, and the section where candidates most often fall back on buzzword-level answers. Aim to describe mechanism, not slogans.

What the GIL actually is

The Global Interpreter Lock is a mutex held by the CPython interpreter loop that serializes access to Python object state. At any moment, exactly one OS thread in a given interpreter can be executing Python bytecode. It does not make individual bytecode operations atomic in any useful sense beyond "one instruction runs at a time" — but that one-at-a-time-ness is enough to make refcounting safe without per-object locks.

The GIL is not a language-level requirement. It's an implementation detail of CPython that the broader ecosystem has come to rely on through decades of C extensions. Jython doesn't have one. IronPython doesn't. PyPy has a GIL but with different semantics. And CPython 3.13 ships an optional free-threaded build (PEP 703) that removes it.

How the GIL is scheduled

Since Python 3.2, the GIL uses a cooperative + preemptive scheme:

  • A running thread holds the GIL until either (a) it blocks on I/O or explicitly releases it, or (b) a configurable interval elapses.
  • The interval is controlled by sys.setswitchinterval(seconds) and defaults to 5 ms. You can read it with sys.getswitchinterval().
  • When the interval ticks, the running thread signals waiters and drops the lock; another thread takes it.

Crucially, C extensions can (and should) release the GIL around long-running C work using the Py_BEGIN_ALLOW_THREADS / Py_END_ALLOW_THREADS macros. NumPy, hashlib, the I/O subsystem, and most of the compression libraries all do this — which is why "Python threads are useless" is overstated. 

import sys
sys.getswitchinterval()     # 0.005
sys.setswitchinterval(0.001)  # aggressive

Atomicity: what is and isn't safe without a lock

A handful of operations are atomic in CPython because they correspond to a single bytecode instruction that doesn't drop the GIL:

  • x = y (simple name binding)
  • L.append(item), D[k] = v, D.get(k) for built-in types
  • L.extend(other), D.update(other)

But operations like x += 1 are not atomic: they compile to LOAD → ADD → STORE, and the GIL can be released between those. Likewise, anything that invokes Python-level methods (including __hash__, __eq__) can yield. The safe default is: assume nothing is atomic, use locks. The exceptions above are for understanding CPython, not for shipping code.

Threads — when they help

Threads are the right answer whenever the bottleneck is outside Python: blocking syscalls, database drivers, HTTP clients, or C extensions that release the GIL. They're also fine for small amounts of background work. They're the wrong answer for CPU-bound Python code — you will see negative scaling because of GIL contention plus context-switch overhead. 

import threading, queue, requests

q = queue.Queue()
def worker():
    while (url := q.get()) is not None:
        try:
            q.put_result(requests.get(url, timeout=5))
        finally:
            q.task_done()

threads = [threading.Thread(target=worker, daemon=True) for _ in range(16)]
for t in threads: t.start()

asyncio — the model

asyncio is a single-threaded cooperative scheduler over a non-blocking I/O multiplexer (epoll / kqueue / IOCP). A coroutine is a resumable function; await yields control back to the loop; the loop wakes coroutines when their awaited futures complete. Under the hood, a Task is the scheduler-visible handle that drives a coroutine forward. 

import asyncio

async def fetch(name, delay):
    await asyncio.sleep(delay)
    return name

async def main():
    tasks = [asyncio.create_task(fetch(f"t{i}", i * 0.1)) for i in range(5)]
    results = await asyncio.gather(*tasks)
    print(results)

asyncio.run(main())

Four things to drill:

  1. async def f(): returns a coroutine object without running it. You need await, asyncio.run, or create_task to actually drive it.
  2. A coroutine that blocks (a sleep that doesn't use asyncio.sleep, a sync DB call, a time.sleep) stalls the entire loop. This is the "don't block the event loop" rule. Use loop.run_in_executor to offload blocking work to a thread or process pool.
  3. asyncio.gather schedules concurrently and returns a list; asyncio.wait gives you more control with first-complete semantics; asyncio.TaskGroup (3.11+) is the structured-concurrency version that cancels siblings on failure.
  4. Exceptions in unawaited tasks will be swallowed at GC time — always use TaskGroup or explicit await/exception().

multiprocessing — when you really need cores

multiprocessing sidesteps the GIL by running separate Python processes and coordinating with IPC. The price is real:

  • Startup: fork is fast but shares memory copy-on-write and doesn't play well with threads; spawn is slower but portable and safer; forkserver is a middle ground. Pick spawn if you will be running on macOS or alongside threads.
  • Serialization: arguments and return values cross process boundaries via pickle. Large objects are a tax.
  • Shared memory: multiprocessing.shared_memory.SharedMemory and Manager let you share state, but neither is free. For NumPy arrays, shared_memory-backed arrays are a common pattern to avoid copying. 
from concurrent.futures import ProcessPoolExecutor

def work(chunk):
    return sum(x * x for x in chunk)

with ProcessPoolExecutor(max_workers=8) as ex:
    results = list(ex.map(work, chunks))

Free-threaded CPython (PEP 703)

CPython 3.13 ships an opt-in build where the GIL is removed. The techniques used to keep refcounting correct without a global lock:

  • Biased reference counting — each object has an "owning" thread; refcount updates from the owner are non-atomic, others use atomic operations.
  • Deferred reference counting for objects on the interpreter's internal stack.
  • Per-type locks around mutations of built-in containers.

In the transitional period (3.13/3.14), most pure-Python code will "just work" but you will pay a ~10% single-threaded overhead. C extensions need to opt in to free-threading and may require audits. For CPU-bound analytic code this is a game-changer; for I/O or glue code, asyncio/threads are usually already fine.

Subinterpreters (PEP 684)

Since 3.12, you can run multiple interpreters in a single process, each with its own GIL. They share OS resources but don't share Python-level state, and cross-interpreter communication goes through queues and channels. The most immediate benefit: multiprocessing-style parallelism without the fork/spawn cost. The catch today is C extension compatibility — many extensions assume a single interpreter and will crash if loaded in multiple.

Pitfall
"The GIL means Python can't do parallelism" is wrong. The correct version is: "CPython's GIL prevents Python bytecode from executing in parallel within one interpreter. C extensions routinely release the GIL for their compute, which is why NumPy, SciPy, hashlib, and zlib all see real speedups with threads. And with 3.13's free-threaded build or subinterpreters, even pure Python can scale across cores."

Walk me through exactly what the GIL is and why it exists.

The GIL is a mutex in the CPython runtime that is held whenever a thread is executing Python bytecode. It exists because CPython's memory model is reference-counted, and making every Py_INCREF and Py_DECREF atomic would add a measurable tax on every single pointer operation. The GIL sidesteps that by serializing bytecode execution so refcount updates are implicitly safe. A thread holds the GIL until either it blocks on I/O, it calls a C function that explicitly releases it, or it hits the switch interval — 5 ms by default. The tradeoff is that two CPU-bound Python threads do not run in parallel on two cores in the default build.

When would you use threads vs asyncio vs multiprocessing?

Short version: threads for blocking I/O and C libraries that release the GIL; asyncio for high-concurrency network code where you want tens of thousands of concurrent tasks with low per-task overhead; multiprocessing for CPU-bound Python code that can't be vectorized into NumPy or moved into a C extension. Longer version: threads have lower per-task overhead than processes but don't help with CPU-bound Python; asyncio has the lowest per-task overhead but poisons the whole codebase with async coloring and dies the moment something blocks the loop; multiprocessing actually parallelizes CPU work but pays pickling and startup costs. If you're reaching for multiprocessing for numeric work, also ask whether NumPy, Cython, or numba solves the problem without leaving one process.

I want to share a large NumPy array across worker processes without copying. How?

Three common options. First, if you're on Linux with fork start method, the child inherits the parent's pages copy-on-write — as long as nothing writes, there's no copy. This is fragile because any attribute access on a Python object bumps its refcount, which does a write. Second, multiprocessing.shared_memory.SharedMemory gives you a named shared buffer you can back a NumPy array by, with explicit lifetime management. Third, memory-map a file with numpy.memmap — this also works across unrelated processes and is what a lot of research pipelines use for tick data. For true zero-copy between Python processes, shared_memory + ndarray is the cleanest modern answer.

My asyncio server's latency spikes periodically. What would you investigate first?

Three usual suspects. First, something is blocking the event loop — a synchronous library call, a heavy json.loads, or CPU work inside a coroutine. I'd enable loop.set_debug(True) and PYTHONASYNCIODEBUG=1, which logs callbacks that exceed the configured threshold, and run py-spy with --idle to see where time is going. Second, GC pauses on a large heap — check gc.callbacks, or disable and manually invoke GC at safe points. Third, the scheduling interval on upstream threads or the switch interval itself; if there's cross-thread work, the GIL handoff adds jitter. I'd rule out the loop-blocking case first because it's by far the most common.

Is dict[k] += 1 thread-safe in CPython?

No. It compiles to a load, an add, and a store, and the GIL can be released between them on the switch interval. Two threads racing will lose updates. The canonical fixes are a lock around the increment, or a collections.Counter whose individual updates are each single operations under a lock you hold, or atomic counters from itertools.count if you just need unique integers. Even list.append, which is atomic at the bytecode level, is only safe for the single append — read-then-append sequences are not.

6. Generators, Coroutines, and asyncio Internals

Generators are the hinge around which Python's concurrency story turns. Everything from list comprehensions to asyncio to the iteration protocol is built on the same piece of machinery: a frame you can suspend and resume. Interviewers love this area because the layers are clean and the terminology trips people up.

Iterators, generators, generator expressions

Iteration in Python is a protocol: an iterable has __iter__ that returns an iterator; an iterator has __next__ that returns the next value or raises StopIteration. for x in xs: is sugar for it = iter(xs); while True: try: x = next(it); ... except StopIteration: break.

A generator function (one containing yield) returns a generator object the first time it's called. The generator object implements both __iter__ (returns itself) and __next__. Each next() resumes the function's frame from the last yield, runs until the next yield or return, and either produces a value or raises StopIteration. Local state, the instruction pointer, and the call stack of the generator are all preserved in a frame object on the heap. 

def fib():
    a, b = 0, 1
    while True:
        yield a
        a, b = b, a + b

it = fib()
next(it); next(it); next(it)   # 0, 1, 1

A generator expression — (x * x for x in xs) — is syntactic sugar for an anonymous generator function. It's lazy, doesn't materialize an intermediate list, and composes well with sum, any, all, max, min. Prefer it to list comprehension when the result is consumed once.

yield, send, throw, close

Generators are two-way: values flow out on yield, and values flow back in on send. 

def echo():
    received = None
    while True:
        received = yield received       # yields `received`, waits for next send()
        print("got", received)

g = echo()
next(g)             # prime: advance to the first yield
g.send("hello")     # prints "got hello", yields "hello"
g.throw(ValueError) # inject an exception at the paused yield
g.close()           # raise GeneratorExit at the yield, expected to return
Priming gotcha
You must advance a generator to its first yield before you can send a value — a brand-new generator is paused before the first instruction, not at the first yield. Calling send(None) or next() once is the usual pattern. This bites everyone at least once.

yield from — delegation and the ladder to async

yield from iter is sugar for "yield every value from iter, forwarding sends and throws as if the caller were talking directly to the inner generator, and return the inner generator's StopIteration.value." It collapses what would otherwise be a manual proxying loop. 

def chain(*iterables):
    for it in iterables:
        yield from it

list(chain([1,2], (3,4), "ab"))   # [1, 2, 3, 4, 'a', 'b']

Historically, before native coroutines, yield from was how asyncio composed coroutines. You can still use generator-based coroutines with @asyncio.coroutine, but they're deprecated — prefer async def everywhere.

Native coroutines: async def, await

A coroutine is a generator at the VM level with two differences:

  1. The function is marked with CO_COROUTINE so you can only drive it with await, not plain next().
  2. await expr requires expr to be awaitable — either another coroutine, a Future/Task, or any object with __await__.

Under the hood, await is effectively yield from expr.__await__() plus a marker. A Future's __await__ yields self once and then returns the result, which is how the event loop gets a handle to the task and wakes it when ready. 

import asyncio

async def worker(name, delay):
    await asyncio.sleep(delay)
    return name

async def main():
    async with asyncio.TaskGroup() as tg:         # structured concurrency
        a = tg.create_task(worker("A", 0.1))
        b = tg.create_task(worker("B", 0.2))
    return a.result(), b.result()

asyncio.run(main())

Event loop anatomy

The asyncio event loop is roughly:

  1. A deque of ready callbacks and a heap of scheduled callbacks (keyed by monotonic time).
  2. A selector (wrapping epoll/kqueue/IOCP) that knows about file descriptors of interest.
  3. A main loop: drain the ready queue, call selector with a timeout equal to "time until next scheduled callback," dispatch any FD events into the ready queue, move scheduled callbacks whose time has come into the ready queue, repeat.

A Task is a Future that wraps a coroutine. When you call create_task, the loop schedules task.__step — which calls send(None) on the coroutine. The coroutine runs until it awaits something. Whatever it yielded out is captured as the Task's "currently waiting on" Future; the Task installs a done-callback on that Future that re-schedules __step, then control returns to the loop. The cycle continues until the coroutine returns (setting the Task's result) or raises (setting its exception).

Async iteration and async context managers

async def stream():
    for i in range(5):
        await asyncio.sleep(0)
        yield i       # this is an async generator — async def + yield

async def consume():
    async for x in stream():
        print(x)

class AsyncLock:
    async def __aenter__(self):
        # acquire asynchronously
        return self
    async def __aexit__(self, exc_type, exc, tb):
        # release
        return False

async def use():
    async with AsyncLock() as l:
        ...

Structured concurrency with TaskGroup

Before 3.11 you'd use asyncio.gather — which has a famous bug: if one task fails, the others keep running, and their exceptions can be silently swallowed. TaskGroup fixes this: if any task raises, all siblings are cancelled, and the whole group re-raises an ExceptionGroup. This is the idiomatic modern pattern and worth pattern-matching in interview answers. 

async def fetch_all(urls):
    async with asyncio.TaskGroup() as tg:
        tasks = [tg.create_task(http.get(u)) for u in urls]
    return [t.result() for t in tasks]   # all finished, any failure cancels the rest

Cancellation — the one thing everyone gets wrong

Cancelling a task injects a CancelledError at the next await point. If you catch Exception in a task, you will accidentally swallow CancelledError and the cancel will fail silently. Since 3.8, CancelledError inherits from BaseException, not Exception, which helps — but any broad except BaseException or explicit except CancelledError: pass re-introduces the bug. Rule of thumb: in task bodies, only catch specific exceptions, or re-raise cancellation explicitly.

Talking point
"Native coroutines are generator frames with a different marker bit. await is the moral equivalent of yield from. The event loop is a selector plus a ready queue. That's the whole story — everything else is API."

Why does await need an async def function?

Because await is compiled into bytecode that only makes sense inside a coroutine frame. The async def keyword sets the CO_COROUTINE flag on the compiled code object, which both tells the runtime this function must be driven by send/event loop rather than called directly, and enables the GET_AWAITABLE / YIELD_FROM-like bytecode sequences that await emits. You literally can't emit the right bytecode outside an async def.

What's the difference between a coroutine, a task, and a future?

A coroutine is the raw resumable object returned by calling an async def function — it's inert until something drives it. A Future is a low-level placeholder for a result that will be available later; it has set_result/set_exception/add_done_callback and is what the event loop uses as its scheduling primitive. A Task is a Future subclass that wraps a coroutine and drives it on the event loop: creating a Task schedules the coroutine for execution, and the Task becomes the handle you wait on. You await a Task or Future to get its eventual result; you create a Task to start concurrent work.

Will a generator's finally block always run?

Almost always — close() on a generator raises GeneratorExit at the paused yield, which will propagate through any finally blocks before the generator exits. And the cycle collector calls close() on generators as part of cleanup, so even an abandoned generator eventually runs its finally. The exception is if the interpreter is shutting down — built-in modules can be torn down before your finally runs, which is why you sometimes see "NoneType has no attribute X" from a finally during shutdown. For critical cleanup, use a context manager instead.

What happens if I call time.sleep(1) inside an async function?

You block the entire event loop for a full second. time.sleep is a synchronous C call — it doesn't release the loop, so every other task in this event loop is frozen until it returns. The fix is await asyncio.sleep(1), which yields to the loop and reschedules the coroutine after one second. For unavoidable blocking calls (e.g., a legacy sync library), wrap them with await loop.run_in_executor(None, blocking_call, args) so they run in a thread pool and the loop stays responsive.

Compare asyncio.gather and asyncio.TaskGroup.

Both run coroutines concurrently. The difference is failure semantics. gather returns a list of results (or exceptions if return_exceptions=True); if one task raises, the others keep running and you have to manually cancel them. Worse, exceptions in other tasks can be silently lost if nothing awaits them. TaskGroup implements structured concurrency: on the first failure, all sibling tasks are cancelled, and the group raises an ExceptionGroup containing everything that went wrong. It's the correct default for new code in 3.11+. Reach for gather only when you explicitly want fire-and-forget best-effort semantics.

7. Functions, Closures, Decorators, Context Managers

Functions are objects with a compiled code object, a globals reference, default values, and — if they close over outer scopes — a tuple of cells. Most "what does this print?" interview questions live in the space between these parts.

Function anatomy

def f(a, b=10, *args, key=None, **kwargs):
    return a + b

f.__code__             # compiled code object, immutable
f.__defaults__         # (10,)   positional defaults
f.__kwdefaults__       # {'key': None}   keyword-only defaults
f.__closure__          # None — no closure here
f.__globals__          # dict of the defining module
f.__name__             # 'f'
f.__qualname__         # 'f' or e.g. 'Outer.method' for nested
f.__dict__             # mutable per-function state — yes, functions have their own dict

Closures and cell objects

When a nested function references a name from an enclosing scope, the compiler allocates a cell for that name in the outer frame and puts a reference into the nested function's __closure__ tuple. Both frames point at the same cell, which is how nonlocal rebinding works. 

def make_accumulator():
    total = 0
    def add(x):
        nonlocal total
        total += x
        return total
    return add

acc = make_accumulator()
acc.__closure__[0].cell_contents   # 0, then updates as you call acc()
The late-binding closure trap
fs = [lambda: i for i in range(3)]
[f() for f in fs]   # [2, 2, 2] — all cells point at the same `i`, now equal to 2

The fix is either a default-argument trick (lambda i=i: i, which captures at definition time) or a factory function. This is the single most frequently-asked interview gotcha on closures — be ready.

Decorators — the full vocabulary

A decorator is just func = decorator(func). Any callable qualifies. The patterns you must be fluent in:

Plain function decorator

from functools import wraps

def log(fn):
    @wraps(fn)                      # preserves __name__, __doc__, __wrapped__
    def inner(*args, **kwargs):
        print(f"call {fn.__name__}")
        return fn(*args, **kwargs)
    return inner

@log
def greet(name): return f"hi {name}"

Decorator with arguments

def retry(times):
    def decorator(fn):
        @wraps(fn)
        def inner(*args, **kwargs):
            last = None
            for _ in range(times):
                try: return fn(*args, **kwargs)
                except Exception as e: last = e
            raise last
        return inner
    return decorator

@retry(times=3)
def flaky(): ...

Class decorators and decorator classes

class CountCalls:
    def __init__(self, fn):
        self.fn = fn; self.n = 0
    def __call__(self, *a, **kw):
        self.n += 1
        return self.fn(*a, **kw)

@CountCalls
def work(): ...

Class decorators (decorators applied to classes, not decorator classes) are how @dataclass and @total_ordering work: they inspect the class, synthesize methods, and return the (possibly modified) class.

Stacking

@a @b @c def f(): ... is equivalent to f = a(b(c(f))). The bottom-most decorator runs first. Order matters when decorators log, cache, or register — a caching decorator above a logging decorator will make logs disappear on cache hits.

  • suppress(Exception) — local try/except/pass as a context manager.

What does @wraps do and why does it matter?

functools.wraps copies __name__, __doc__, __module__, __qualname__, __dict__, and __wrapped__ from the wrapped function onto the wrapper. Without it, every decorated function claims to be called inner and has no docstring, which breaks introspection, logging, debuggers, and documentation generators. It also sets __wrapped__, which lets inspect.signature follow through decorators to the original signature. It's not optional — treat it as part of the decorator boilerplate.

How would you implement @lru_cache(maxsize=N) from scratch?

Two classic implementations. First: OrderedDict keyed on the tuple (args, frozenset(kwargs.items())) — on hit, move_to_end; on miss, call and insert, evicting with popitem(last=False) if over capacity. Second: a dict plus a doubly linked list of nodes. Dict maps key to node; the list tracks recency; on access, unlink and push to the head; on eviction, pop the tail. Both are O(1) per op. The stdlib implementation is the second pattern, written in C for speed. Worth mentioning: the key must be hashable, and the cache holds strong references — so caching methods keeps the whole instance alive, which is why people reach for cached_property or weakref.WeakValueDictionary instead.

Why does @property make an attribute un-shadowable by an instance assignment, while a plain method is shadowable?

Because property is a data descriptor (it defines __set__) and plain functions are non-data descriptors (only __get__). The attribute lookup algorithm gives data descriptors priority over instance dicts, so obj.price always calls the property getter even if you assigned obj.price = 5; in fact, that assignment calls the property's __set__, which raises if there's no setter. For a plain method, the lookup finds a non-data descriptor after checking the instance dict, so assigning a function to the instance dict shadows the class method.

8. Typing, Dataclasses, and Modern Declarative Python

Python's type system is gradual: annotations are optional, erased at runtime (mostly), and verified by an external tool like mypy or pyright. Modern quant Python is heavily typed because it's the cheapest way to catch bugs in a dynamic language — expect interviewers to probe how well you use the type system, not whether you use it.

The big picture

  • Annotations are stored in f.__annotations__ (or cls.__annotations__). They're evaluated at function/class definition time by default — this is why forward references need strings or from __future__ import annotations.
  • The type system distinguishes nominal (inheritance-based) from structural (protocol-based) subtyping. Protocol classes are the Python answer to duck typing with teeth.
  • Covariance, contravariance, and invariance matter: list[Dog] is not a list[Animal] (invariant), but Sequence[Dog] is a Sequence[Animal] (covariant because read-only).
  • Runtime enforcement is opt-in (pydantic, attrs with validators, typeguard). The annotations themselves don't check anything.

How does from __future__ import annotations change things, and why would you use it?

It makes all annotations evaluated lazily as strings at class/function definition time rather than as real expressions. The upsides: forward references become free (you can reference a class before it's defined), circular imports become less fragile, and class-definition overhead drops. The downsides: anything that reads annotations at runtime (pydantic, dataclasses in some cases, typing.get_type_hints) has to resolve them with get_type_hints, which requires the referenced names to be importable. It was slated to become the default under PEP 563, but that was deferred; many codebases still turn it on explicitly and it's a good default for library code.

What's the difference between Protocol and a nominal ABC?

An abc.ABC requires explicit subclassing — a type has to inherit from it to be considered a subtype. A Protocol is structural: any object with the matching attributes and methods is accepted by the type checker, no inheritance needed. Protocols are Python's answer to duck typing in a type system — you can type "anything with a price attribute" without forcing downstream code to import and subclass your class. Protocols also support @runtime_checkable for isinstance, but that's slower and loses the structural-typing benefits for static checkers.

When would you reach for pydantic over @dataclass?

When the data crosses a trust boundary and you need validation and coercion. Config loading, API payloads, CLI argument parsing, message queue consumers — anywhere you don't control the upstream. pydantic will validate types, coerce when possible (string to datetime, int to float), run custom validators, and give you rich error messages. For internal record types where you just want a typed struct, @dataclass(slots=True, frozen=True) is cheaper at both runtime and cognitive load. In a latency-sensitive path, neither is ideal — msgspec gives you validation at C speeds.

When should I use deque instead of a list?

Whenever you need fast inserts or pops from the left. list.append and list.pop() from the right are amortized O(1), but list.pop(0) and list.insert(0, x) are O(n) because every element shifts. deque is a doubly linked list of blocks, so both ends are O(1). Common uses: BFS frontiers, sliding windows (with maxlen), and producer-consumer queues between threads. You lose fast random indexing — deque[k] is O(n/block_size), so don't use it as a general list replacement.

What's the complexity of Counter.most_common(k)?

O(n log k) — it uses heapq.nlargest internally, maintaining a min-heap of size k across the n elements. If you call most_common() with no argument, it's O(n log n) because it sorts the whole thing. For top-k work on large streams, always pass the k.

10. Gotchas and Antipatterns

Every one of these has been in an interview. Treat the list as a checklist: if you can't explain each in one sentence with the fix, spend extra time on it.

Mutable default arguments

def append(x, xs=[]):    # evaluated ONCE at def time — shared across calls
    xs.append(x); return xs

append(1); append(2)     # [1], [1, 2] — surprise!

Fix: use None and create a fresh object inside: def append(x, xs=None): xs = xs or []. Why it happens: default values are evaluated at function-definition time and stored in f.__defaults__. Every call that doesn't supply the argument reuses the same object.

Late binding in closures

fs = [lambda: i for i in range(3)]     # all return 2

Cells bind names, not values. Fix: lambda i=i: i captures by default at definition, or use a factory: [(lambda v: lambda: v)(i) for i in range(3)].

Integer identity at the cache boundary

a, b = 256, 256;  a is b    # True — cached
a, b = 257, 257;  a is b    # possibly False

Don't use is for value comparison. is is for identity, singletons, and sentinels (x is None, x is _missing). The small-int cache is [-5, 256], and string interning applies to identifier-like short strings.

bool is an int

isinstance(True, int)   # True
True + True             # 2
sum([True, False, True]) # 2 — which is occasionally useful

This is why Counter({"a": True}) does not do what you might think, and why 1 == True is True. Be careful when a function accepts "a number or a flag."

Tuples of mutables — += weirdness

t = ([], [])
t[0] += [1]          # TypeError: 'tuple' object does not support item assignment
                     # ...but the list WAS extended
t                    # ([1], [])

Why: += on a list calls __iadd__ which mutates in place and returns the same object, then the assignment to t[0] happens, which fails because tuples are immutable. Net effect: the list changed and the assignment raised. Don't use += on mutable elements of immutable containers.

Dict mutation during iteration

Adding or removing keys while iterating raises RuntimeError: dictionary changed size during iteration. Materialize first: for k in list(d): ....

Shadowing built-ins

list = [1, 2, 3], dict = {"a": 1}, id = 5 — all legal, all silently shadow the built-in in the local scope. Linters catch most of these; don't disable them.

Overly broad except

try:
    work()
except:              # catches everything, including SystemExit and KeyboardInterrupt
    pass

Use except Exception: at the absolute broadest. Better: catch the specific exceptions you expect. In asyncio, except Exception is the right ceiling because CancelledError is a BaseException — catching Exception won't accidentally swallow cancellation.

Mutable class attributes

class Bag:
    items = []        # shared across ALL instances
    def add(self, x): self.items.append(x)

a, b = Bag(), Bag()
a.add(1); b.items   # [1] — shared!

Fix in __init__, or use a dataclass with a default_factory.

Chained comparisons

0 < x < 10 is equivalent to 0 < x and x < 10, which is almost always what you want. But a < b < c == d means a < b and b < c and c == d — read carefully in non-trivial cases.

Walrus operator scope

:= binds in the enclosing scope, not the comprehension scope. This is deliberately useful, but surprising: 

[y := x * 2 for x in range(3)]
y   # 4 — leaked out of the comprehension

Circular imports

Circular imports fail at attribute access time, not import time. If a.py does from b import X and b.py does from a import Y, the first one to start importing will populate its own module object, the second import will find a half-initialized module and likely fail to find the attribute. Fixes: import inside the function that needs it (lazy), restructure so the dependency is one-way, or push the shared code into a third module.

Async pitfalls

  • Forgetting to await returns a coroutine object that never runs. 3.8+ warns about unawaited coroutines.
  • Blocking calls in coroutines — time.sleep, requests.get, file I/O on a spinning disk, heavy json.loads on large strings. All stall the loop.
  • Fire-and-forget tasks without keeping a reference — the task can be garbage-collected mid-execution. Always keep the Task object in scope, or use a TaskGroup.
  • Broad except Exception inside tasks can swallow CancelledError in older versions; use structured concurrency and catch specifics.

__del__ and finalizer quirks

  • Runs at an unspecified time (or never, during interpreter shutdown).
  • Exceptions are printed and swallowed — you will not see them in production unless you look.
  • Can resurrect its own object by adding a strong reference somewhere.
  • Historically blocked cycle collection when combined with cycles (fixed since 3.4, but still a code smell).

Use weakref.finalize or a context manager instead.

Surprising equality and hashing

  • Overriding __eq__ sets __hash__ to None automatically — objects become unhashable unless you also define __hash__.
  • NaN != NaN, but NaN in [NaN] returns True because containers use identity for membership as a fast path.
  • hash(-1) is -2 — -1 is reserved to mean "error" in CPython.

11. Pros and Cons of Python

Expect the meta-question: "Why do we even use Python here?" or "What would you say Python is bad at?" A good answer names tradeoffs precisely and connects them to the work.

Pros — why Python wins in quant and glue code

  • Ergonomics and iteration speed. Dynamic typing plus REPL plus notebook tooling plus live reloading means research ideas move from whiteboard to prototype in minutes. This is almost a competitive advantage in research workflows — other languages force a compile-link-run loop that kills iteration.
  • Batteries included. Stdlib covers most plumbing you'd reach for a third-party library for in other languages: asyncio, multiprocessing, compression, hashing, sockets, argparse, logging, unittest, sqlite.
  • Ecosystem depth. NumPy, SciPy, Pandas, PyArrow, scikit-learn, PyTorch, JAX, polars, statsmodels, CVXPY. Nothing in any other language gets close for numeric and scientific work.
  • Great C FFI. ctypes, cffi, Cython, pybind11, nanobind — any performance-sensitive piece can live in C/C++/Rust with a thin Python wrapper. This is how Python punches above its weight: the hot path isn't Python.
  • Type hints have matured. Modern Python with mypy/pyright is close to a statically-typed language for the purposes of catching bugs in IDEs and CI, without losing the dynamism for research code.
  • Readability. It's not a meme — code reviews move faster in Python than in most languages, and a research scientist can read and modify a trading engineer's Python code (and vice versa).
  • Polyglot glue. Python talks to everything — C extensions, Rust via PyO3, Java via JPype, DBs, message queues, arbitrary command-line tools. This is the "glue language" argument, and at a firm with a C++ trading core, the gluing is the job.

Cons — where Python hurts

  • Single-threaded CPU performance. The GIL plus interpretation overhead plus heap-allocated everything means naive Python is one to two orders of magnitude slower than C for compute. Mitigated by NumPy, Cython, numba, PyPy, and now free-threaded CPython — but it's a real limitation you have to plan around.
  • Memory overhead. Every object has a 16+ byte header plus alignment; lists are pointer arrays; dicts have load-factor slack. A list of ints uses 4x the memory of a C array. For millions of small records, __slots__, NumPy structured arrays, or msgspec are near-mandatory.
  • Startup time. Even trivial CLIs take 50-100 ms to start because of imports. For short-lived subprocesses or Lambda-style workloads, this dominates everything else. Workarounds: long-lived daemons, PYTHONSTARTUP warmup, Cython-compiled modules.
  • Packaging and deployment. Historically a mess. Has improved substantially with pyproject.toml, pip, uv, pipx, but still more friction than Go's single binary or Rust's cargo. Native dependencies (manylinux wheels) are still a source of pain.
  • Dynamic typing bites at scale. Without type hints and mypy in CI, large Python codebases drift into "works until it doesn't." Refactoring is slower than in a statically typed language because the compiler doesn't find all the call sites for you.
  • Mobile / embedded / browser. Not a serious story. Pyodide (WASM) is a research curiosity; mobile Python exists but isn't mainstream.
  • Version fragmentation and 2-to-3 scar tissue. Still occasionally bites when working with old libraries.
  • Async is colored. The async/sync split means you can't casually call a sync library from async code without thread pool offload, and can't call async code from sync code without running an event loop. This colors libraries too — there's a whole parallel ecosystem (httpx, aiopg, etc.).

The honest summary for a quant firm

Python wins because research-to-production latency is the dominant cost in alpha generation, and Python compresses that loop better than any alternative. Its CPU performance is worked around by pushing compute into C/NumPy/Cython or out of process; its packaging is managed with internal tooling; its GIL is routed around with async for I/O and subprocesses for parallelism. You accept the tradeoffs because the alternatives — C++ for research, Java for scripting — would slow iteration more than they'd speed execution.

Interview one-liner
"Python is not fast, but it's fast enough to let smart people iterate on ideas cheaply, and the places it's slow have well-trodden escape hatches. In a latency-sensitive shop, you use it as glue around a C++ core, not as the core itself."

12. Best Practices

Beyond "use type hints" and "run black": the practices a production Python codebase at a trading firm will expect you to reach for by default.

Project layout and packaging

  • pyproject.toml first. setup.py is legacy. A single pyproject.toml declares build system, metadata, dependencies, and tool configuration (ruff, mypy, pytest). It's the 2020s baseline.
  • Src layout. Put code under src/your_package/ rather than at the repo root. Prevents accidental implicit imports and catches "works because I'm in the repo" bugs.
  • Lockfiles. Use uv, poetry, or pip-tools to lock exact transitive versions. Unlocked dependencies are a time bomb.
  • Pinned Python. Your tooling and runtime should agree on the interpreter version. Mismatches are a rich source of "works on my machine."

Typing and linting

  • Type hints everywhere they buy something. Public APIs, dataclass fields, function signatures. Local variables usually don't need them — the checker infers.
  • Pyright or mypy in CI, configured strict-ish for new code. For legacy code, use mypy --strict-optional at a minimum.
  • ruff is the modern fast linter — replaces flake8, isort, pyupgrade, and a dozen others in a single tool. Configure it in pyproject.toml.
  • black or ruff-format for auto-formatting. No bikeshedding on style in PRs.

Testing

  • pytest is the standard. Fixtures over setUp/tearDown. Parametrize ruthlessly.
  • hypothesis for property-based testing — surprisingly valuable for parsers, state machines, and anything mathematical.
  • pytest-benchmark for tracking performance regressions on critical functions.
  • Integration tests that exercise real dependencies (DBs, queues) are worth the extra runtime. Mocked tests catch the bugs you wrote and miss the ones you didn't.
  • Keep unit tests fast (<1 s per file) so people run them. If they're slow, they won't run them.

Error handling

  • Catch narrowly. except FileNotFoundError, not except Exception. Re-raise with context using raise X("...") from original.
  • Don't use exceptions for control flow in hot paths. Raising costs several microseconds in 3.12.
  • Fail loud, fail early. Log the exception with traceback at the boundary (top-level request handler, task boundary), not in every function.
  • Use logging, not print. Configure once at program start; use module-level logger = logging.getLogger(__name__). Structured logging (JSON with fields) is the default for production services.

Performance discipline

  • Measure, don't guess. cProfile for function-level, line_profiler for per-line, py-spy for sampling a live production process without restarting it, scalene for memory + CPU together.
  • Profile before optimizing. The bottleneck is almost never where you think it is.
  • Benchmark with realistic data. A microbenchmark with 100 elements tells you nothing about behavior at 10 million.
  • Prefer data structures over cleverness. The right set, dict, or deque beats a clever algorithm on a suboptimal structure every time.

Concurrency hygiene

  • Don't share mutable state across threads. Use queues.
  • Don't block the event loop. Offload sync work with run_in_executor.
  • Use structured concurrency. TaskGroup over gather whenever possible.
  • Set explicit timeouts on every external call. No exceptions.

Code that reviews well

  • Functions do one thing. If you can't name it in a short phrase, split it.
  • Dataclasses for data. Don't pass bare dicts or tuples across module boundaries.
  • Immutable by default. Frozen dataclasses, tuples, @cache instead of stateful memoization.
  • Docstrings on anything public. At minimum, what it does and what the parameters mean.
  • No bare pass in except. Either handle it or don't catch it.

13. Performance and the Quant/HFT Lens

This is the section latency-focused interviewers care about most. A strong answer connects specific CPython behavior to concrete techniques, and — critically — knows when to leave Python entirely.

Why Python is slow (in terms you can defend)

  1. Interpretation overhead. Every bytecode operation goes through the ceval dispatch loop — a switch (or computed goto) over opcodes, with decoding and argument fetching. A single C-level addition becomes dozens of C instructions in the interpreter.
  2. Dynamic dispatch. Every operation must look up a type, find a slot, and call through a pointer. Even with the 3.11+ specializing interpreter caching the lookup, the dispatch cost is non-zero.
  3. Heap allocation per value. Every int, float, and intermediate result is a heap-allocated PyObject. The allocator is fast but not free — and pointer chasing through object headers trashes the CPU cache.
  4. Boxing of numerics. 3.14 is not a register-resident double; it's an 8-byte object header plus a double, accessed through a pointer. A tight numerical loop spends most of its time in box/unbox, not in arithmetic.
  5. The GIL. Covered in §5. Prevents scaling pure-Python compute across cores in the default build.
  6. No ahead-of-time type specialization. The compiler can't assume x is an int, so it can't inline int.__add__ or elide the typecheck. The 3.11+ specializing interpreter does this at runtime, but only within a single frame.

Where to get speed without leaving Python

1. NumPy — the default answer

NumPy arrays are a contiguous block of homogeneous C values with a type descriptor. Operations on them dispatch to pre-compiled C loops that the interpreter calls once. A million additions becomes one C loop call instead of a million bytecode iterations. This is usually 100-1000x faster than the naive Python loop and is the single highest-leverage optimization available. 

import numpy as np

# Python loop: ~500 ms for 1M elements
py = [a * b for a, b in zip(xs, ys)]

# NumPy: ~1 ms
xs_np = np.asarray(xs);  ys_np = np.asarray(ys)
np_result = xs_np * ys_np

Key NumPy performance ideas:

  • Vectorize. Replace loops with array operations. If you can't, you probably want np.vectorize (convenience, not speed) or Cython.
  • Broadcasting. Aligns shapes without materializing — a[:, None] - b[None, :] computes a distance matrix without a Python loop.
  • Views, not copies. Slicing returns a view; arr.copy() forces a copy. Fancy indexing (boolean or integer array) always copies.
  • Dtype discipline. Use float32 when you don't need the precision — half the memory, often faster. Mixing dtypes triggers copies.
  • Avoid object arrays. They're NumPy in name only — just a boxed PyObject array with no speedup.
  • Use np.einsum for contractions; it's surprisingly fast and expressive.
  • Pre-allocate outputs. np.empty + in-place ops (out= kwarg) save allocations in hot loops.

2. Pandas — with caveats

Pandas is a heavy abstraction on top of NumPy. For tabular work it's fantastic; for hot paths it's often surprisingly slow. Modern alternatives:

  • Polars — Rust-backed, columnar, lazy execution, multi-core. Often 5-30x faster than Pandas for real workloads, with a cleaner API. Increasingly the default for research pipelines.
  • PyArrow — the underlying columnar format. Zero-copy across process boundaries, great interop with Parquet.
  • DuckDB — embedded analytical database. Query Parquet or in-memory dataframes with SQL, very fast.

If you're on Pandas, push work into vectorized ops, avoid iterrows and apply, use categorical dtypes for low-cardinality strings, and be aware of the new Arrow backend (dtype_backend='pyarrow') which is significantly faster for string-heavy data.

3. Numba — JIT for numeric Python

from numba import njit

@njit(cache=True)
def ma(x, n):
    out = np.empty_like(x)
    s = 0.0
    for i in range(len(x)):
        s += x[i]
        if i >= n: s -= x[i - n]
        out[i] = s / min(i + 1, n)
    return out

Numba compiles annotated Python to LLVM at first call — typically matching C for numeric loops on NumPy arrays. Limitations: numeric Python subset only, no objects or dynamic features, startup tax on first call. Best for moderate-complexity numeric kernels you'd otherwise write in Cython.

4. Cython — if you need total control

Cython is Python-with-types compiled to C. It's the escape hatch when you need every cycle, can accept a build step, and are willing to read C backtraces. Patterns: cdef typed locals for speed, cimport numpy for direct array access, with nogil: blocks to release the GIL and run multithreaded numeric code.

5. C extensions via pybind11 / nanobind

When the kernel is already C++ (the common case at latency-sensitive firms), wrap it with pybind11 or nanobind. This is the production pattern at most large quant firms. Your Python code becomes a thin glue layer; the arithmetic happens in C++ with all the latency benefits.

6. mypyc and Cython pure-Python mode

mypyc (from the mypy project) compiles type-annotated Python to C extensions directly — no separate syntax. It's a middle ground: small speedups with zero code changes if your types are accurate. Used in production by mypy itself, and some parts of the Dropbox codebase.

Low-latency Python patterns

When you're stuck writing Python in a latency-sensitive path, these patterns are worth internalizing:

  • Preallocate everything. Loops that grow lists or dicts in the hot path are slow. Allocate once, reuse.
  • Hoist attribute lookups. append = list.append inside a loop turns attribute access into a local variable read — measurably faster in tight loops.
  • Local variables beat globals. LOAD_FAST is faster than LOAD_GLOBAL. Make module-level functions parameters when you're calling them millions of times.
  • Avoid exceptions in the hot path. Use explicit checks or sentinels.
  • Use __slots__ for frequently instantiated classes. Not free, but close.
  • Batch. One C call on 10,000 elements is much faster than 10,000 C calls on one element.
  • Avoid isinstance / type() checks in loops when possible — they're not free.
  • Disable GC in very tight phases if you can guarantee no cycles.
  • Use array.array or NumPy for homogeneous numeric storage instead of list.
  • Memoryviews and the buffer protocol give zero-copy slicing over bytes/ctypes/NumPy.
  • Warm up the specializing interpreter. The first few calls to a function are slower than subsequent ones because specialization hasn't kicked in. Warm up the hot functions at startup. 
# Example: hoist, localize, preallocate
def process(ticks, out):
    bid_get = ticks.__getitem__      # hoist
    local_round = round              # localize the builtin
    for i in range(len(ticks)):
        out[i] = local_round(bid_get(i), 4)
    return out

PyPy vs CPython

PyPy is a separate implementation of Python with a tracing JIT. On pure-Python workloads it's typically 3-10x faster than CPython. Why it's not the universal answer: the C extension story is weaker (cpyext is a compatibility shim that's slower than native CPython C extensions), startup is slower, and memory usage is higher. It's the right answer for long-running pure-Python services with heavy computation; it's the wrong answer when you live on NumPy / Pandas / C extensions.

Quant framing
Every real Python performance win eventually looks like "stop running Python in the hot path." NumPy does it by batching. Cython and pybind11 do it by generating C. Shared memory and PyArrow do it by removing serialization. Know which technique maps to which bottleneck, and measure before picking.

Why is a NumPy sum so much faster than a Python sum?

Two reasons. First, the Python sum runs a bytecode loop over the sequence, performing a BINARY_ADD on each element — tens of interpreter instructions per value, plus a heap allocation for each intermediate result. Second, NumPy's sum dispatches once into a compiled C loop that walks a contiguous block of float64 values in place, with tight CPU-level arithmetic and no boxing. On a million doubles, the difference is roughly 100-500x, depending on cache behavior. It's the canonical example of why vectorization matters: you're not just avoiding the bytecode, you're avoiding the entire object model.

How would you speed up a Python function that processes market ticks and is the bottleneck?

Profile first — py-spy record on a realistic run, then read the flame graph. Then, in order of effort: (1) see if the inner loop can be vectorized with NumPy; (2) check for avoidable allocations and hoist repeated attribute lookups; (3) __slots__ on the hot data class; (4) if the logic doesn't vectorize cleanly, rewrite the kernel in numba or Cython; (5) if numba isn't enough, push the kernel into C++ via pybind11. At each step I'd rerun the profiler to confirm the change actually moved the needle. The biggest wins usually come from either removing a per-tick Python object or replacing a Python loop with a C loop, and the order of attempt is roughly increasing engineering cost.

How do you pass a large NumPy array to a worker process without copying it?

Use multiprocessing.shared_memory.SharedMemory in combination with numpy.ndarray(shape, dtype, buffer=shm.buf). The producer allocates a named shared-memory block, wraps it in an ndarray, fills it, and passes the name to the worker. The worker opens the same block and wraps it in its own ndarray view — both processes are looking at the same bytes. Alternative for on-disk data: numpy.memmap over a file, which lets unrelated processes share via the filesystem page cache. Both avoid the pickle cost you'd otherwise pay on every Queue.put.

What's the difference between vectorization in NumPy and SIMD vectorization in a compiler?

They're related but not the same thing. NumPy vectorization means "replace a Python loop with a single C loop call" — the win comes from not running the interpreter, even if the underlying C loop is scalar. Compiler SIMD vectorization means "generate AVX or NEON instructions that process multiple values per cycle" — the win comes from wider CPU instructions. NumPy's C loops are often (but not always) SIMD-vectorized by the underlying BLAS implementation (OpenBLAS, MKL). When you hear "Python vectorization," it's almost always the first sense. When you hear about bypassing NumPy with Cython or numba, sometimes you get the second sense as a bonus from LLVM.

A final pass of cross-cutting questions of the kind that show up mid-interview once rapport is established. Practice answering these out loud; aim for 60-90 seconds each, with a crisp headline followed by one layer of depth.

Internals & model

What actually happens when I call a.b.c three times in a row?

Each lookup is an independent attribute-access algorithm. a.b looks up b on type(a) and its MRO checking for data descriptors first, then a.__dict__, then non-data descriptors, then type(a).__getattr__. The result is a new reference. .c then runs the same algorithm on that result. Each step can trigger arbitrary Python code (property getters, __getattribute__ overrides, descriptor __get__) and bumps refcounts. In a hot loop, x = a.b.c is measurably slower than binding bc = a.b.c once and reusing bc.

Explain the difference between is and ==.

is compares object identity — are these two names pointing at the exact same object in memory? It's a single C pointer comparison. == dispatches to __eq__, which can be arbitrary Python code comparing values. Use is only for singletons and sentinels: None, True, False, and custom MISSING markers. Using is for value comparison is a common bug that happens to work for small ints and interned strings and then fails in production on larger values.

Why is dict ordered since 3.7?

CPython switched to a "compact dict" representation in 3.6 as an implementation detail, and 3.7 elevated insertion-order preservation to a language guarantee. The layout is a dense array of entries in insertion order plus a sparse hash table of indices into that array. Benefits: lower memory (the sparse table holds int indices, not full entries), better cache locality on iteration, and — as a side effect — ordered iteration. The hash table still resolves collisions with open addressing and perturbs the probe sequence based on the upper bits of the hash.

How does CPython handle hash collisions in dict?

Open addressing with a perturbation-based probe sequence. When a slot collides, the probe function combines the hash and a counter to produce the next index, mixing in high-order hash bits to avoid clustering. The load factor is kept around 2/3 — when exceeded, the table is resized (typically doubling). Deletion marks slots as "dummy" rather than empty so later probes continue. This avoids the cluster-degradation you'd get with linear probing. For adversarial hash inputs, Python uses a randomized hash seed (per interpreter start, unless PYTHONHASHSEED is fixed) to prevent denial-of-service attacks via hash collisions.

What's a metaclass and when would you actually use one?

A metaclass is the class of a class. type is the default metaclass — it's the thing that runs when you execute class Foo: .... By setting metaclass=MyMeta, you get hooks to customize class creation: __new__ runs before the class body is fully built, __init_subclass__ (often a simpler replacement) runs when any subclass is defined. Real uses: registering plugins at class-definition time, enforcing interface constraints, generating slots or methods from declarative class bodies, ORMs mapping attributes to columns. In modern Python, __init_subclass__ and class decorators cover most of the use cases more cheaply, so reach for a full metaclass only when you genuinely need to intercept class construction itself.

Concurrency & performance

You're writing a market-data ingestor that has to handle ~1M messages per second. Where does Python fit, if at all?

Not in the hot path of the byte-level ingest. At 1M msg/s you have one microsecond per message, which is tighter than a Python function call. The realistic architecture is a C++ (or Rust) ingest loop that parses and dispatches, with Python sitting above it as the control plane, configuration, research interface, and slower-path analytics. You expose the C++ side via pybind11 / nanobind and share tick buffers via memory-mapped or shared-memory NumPy arrays so Python can read snapshots without serialization. Python then handles anything that can run at a few hundred Hz or lower: strategy research, parameter updates, logging aggregation, dashboards. The rule of thumb is "Python is where humans touch the system; C++ is where microseconds matter."

Walk me through how you'd run a simple CPU-bound task across 8 cores in Python.

First, I'd check if the task is vectorizable — if it's array math, NumPy already releases the GIL in most kernels, and thread pool + NumPy is a two-line solution with no process overhead. If it's pure Python, the standard answer is concurrent.futures.ProcessPoolExecutor with max_workers=os.cpu_count() and chunked inputs, using spawn as the start method for safety. I'd pay attention to the pickling cost of inputs and outputs, batching large-enough chunks to amortize the IPC and startup. For a 3.13 free-threaded build, or if the kernel is mostly NumPy with some Python glue, I'd just use a thread pool and measure. The anti-pattern is reaching for multiprocessing on a task that fits in one NumPy call.

What concurrency primitives does Python give you, and when do they differ from their equivalents in lower-level languages?

The threading module gives Lock, RLock, Semaphore, Event, Condition, Barrier, Queue. These are backed by OS primitives but acquired under the GIL, so uncontended acquisition is very fast. The one that surprises people: threading.Lock is not reentrant — same thread acquiring twice deadlocks. Use RLock if you need reentry. asyncio mirrors the API with asyncio.Lock, asyncio.Semaphore, etc., but they only work inside the event loop. Different in spirit from C++ or Rust: Python's GIL means you rarely need fine-grained locking because most operations serialize already, so the common mistake is over-locking. The other common mistake is under-locking read-modify-write sequences on shared state, which are not atomic despite the GIL.

How do you avoid GIL contention when calling into a C library that takes a long time?

The C function should release the GIL around its work using the Py_BEGIN_ALLOW_THREADS / Py_END_ALLOW_THREADS macro pair. While those macros are in effect, other Python threads can run freely. Caveats: you cannot touch Python objects while the GIL is released, and you must re-acquire before returning anything to the caller. NumPy, BLAS, hashlib, zlib, and most of the stdlib I/O subsystem already do this. If you're calling a library that doesn't, and you control the wrapper, you can drop the GIL explicitly in your pybind11 / ctypes shim. This is also the prerequisite for threads to actually help with C-bound work.

Language design and judgment

What's your mental model for when to use a dict vs a dataclass vs a NamedTuple?

Dicts for dynamic or schema-less data where keys are not known at write time — config overlays, JSON payloads, intermediate aggregations. Dataclasses for named, typed records with a fixed shape — the 90% default for domain modeling. Use @dataclass(slots=True, frozen=True) when you want memory-efficient, hashable, immutable values. NamedTuples when you also need tuple indexing and unpacking (e.g., interoperating with code that expects tuples) or when you really need immutability without the dataclass decorator machinery. For runtime validation of external data, reach for pydantic or msgspec instead. The biggest anti-pattern is passing bare dicts across module boundaries — they're un-typed, un-IDE-discoverable, and every consumer re-implements the "did the key exist" check.

What does "Pythonic" mean to you?

Functionally: code that takes advantage of the language features rather than transliterating idioms from another language. Concretely, that means using comprehensions and generator expressions instead of manual loops that append; using context managers for resource lifetime; using the iteration protocol and yield instead of state machines; using the stdlib collections rather than reinventing them; using duck typing and protocols over deep inheritance hierarchies; using exceptions for exceptional conditions and EAFP for expected happy paths; and using type hints and docstrings to make intent legible. Culturally, the Zen of Python — "flat is better than nested," "explicit is better than implicit" — captures the aesthetic. Anti-Pythonic code is usually recognizable as Java or C++ translated literally: getter/setter boilerplate, deep inheritance, manual iterator classes, checking types with isinstance everywhere.

If you could change one thing about Python, what would it be?

A sincere answer that also signals depth: I'd want the GIL removed cleanly — which is in flight with PEP 703 — but more importantly, I'd want the packaging story unified. The fragmentation between pip, conda, poetry, pip-tools, pdm, and uv has cost more engineering hours in the Python community than any other single issue, and it's improving but still painful. A less political answer: faster startup. For CLI tools and serverless, 50-100 ms is a real cost, and most of it is module import machinery that could be short-circuited with better caching or AOT-compiled frozen modules. An honest answer you can defend beats a safe answer every time.

What are you most excited about in modern Python?

Three things I'd name. First, the specializing interpreter and the incoming JIT — CPython is finally getting meaningfully faster on real workloads without requiring a rewrite. Second, free-threaded Python in 3.13 — the GIL has been the single biggest architectural constraint for a decade and removing it opens up concurrency designs that were impossible before. Third, the type system: Self, ExceptionGroup, the new generic syntax (PEP 695), and the ever-improving pydantic / msgspec ecosystem are turning Python into something you can ship at scale with real confidence. Honorable mention: polars and PyArrow are changing how data science Python is written.

Behavioral and code-quality

How do you approach reviewing someone else's Python code?

I read it in three passes. First pass for correctness: does it actually do what the PR claims, and does it handle the obvious edge cases — empty inputs, None, type coercion, failure modes. Second pass for structure: is the function doing one thing, are the names honest, is the control flow readable without a map, are the abstractions at consistent levels. Third pass for risk: does it share mutable state, leak resources, broaden an exception net, introduce a silent dependency, create latency in a hot path, depend on undocumented behavior. I try to leave reviews that say why I'm concerned, not just what to change — the goal is for the author to catch similar issues on the next PR without me.

Interviewers want to see that you treat performance as a measurement discipline, not as pattern-matching on buzzwords.
